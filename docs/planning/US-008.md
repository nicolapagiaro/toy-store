## User Story

**Epic:** EP-003 — Scambio, Chat e Fiducia
**Priorità:** HIGH | **Story Points:** 5

**Story**
As genitore interessato a un annuncio,
I want avviare una chat legata direttamente all'annuncio selezionato,
so that posso discutere dettagli senza ambiguita.

**Criteri di Accettazione**
- [ ] Dal dettaglio annuncio posso avviare una chat con il proprietario.
- [ ] Ogni chat mostra riferimento esplicito all'annuncio associato.
- [ ] Un utente non autenticato non puo avviare chat e viene invitato ad autenticarsi.

---

## Soluzione Tecnica

La soluzione introduce la chat contestuale in quattro aree distinte — modello dati, service layer, server action e interfacce — seguendo i pattern di dependency injection e server component già consolidati nel progetto.

- **Modello dati:** due nuovi modelli Prisma — `Conversation` (campi: `id`, `toyAdId` FK→ToyAd, `buyerId` FK→User, `ownerId` FK→User, `createdAt`) con unique constraint su `(toyAdId, buyerId)`, e `Message` (`id`, `conversationId` FK→Conversation, `senderId` FK→User, `content`, `createdAt`). La unique constraint garantisce idempotenza a livello DB senza logica applicativa ridondante.
- **Service `src/lib/chat/service.ts`:** funzioni `getOrCreateConversation` (idempotente, valida buyer ≠ owner, valida esistenza annuncio) e `sendMessage` + `listMessages`, con dependency injection identica al pattern di `src/lib/toy-ads/service.ts`.
- **Server action `startChatAction`:** chiama `getOrCreateConversation`, poi esegue `redirect('/dashboard/chats/[id]')`. Blocca utenti non autenticati con errore esplicito prima di chiamare il service.
- **Pagina pubblica `/ads/[id]`:** server component accessibile a tutti. Mostra i dati dell'annuncio; per utenti autenticati non-proprietari espone il form con `startChatAction`; per non autenticati mostra link "Accedi per contattare" verso `/auth/signin`; per il proprietario mostra link alla propria dashboard.
- **Pagina chat `/dashboard/chats/[id]`:** protetta dal middleware esistente (`/dashboard/*`). Recupera la conversazione verificando che l'utente corrente sia buyer o owner (`notFound()` altrimenti). Layout: card di riferimento annuncio (titolo, prima immagine, città) + lista messaggi + form di invio con server action `sendMessageAction`.

## Strategia di Test

I test seguono il pattern Vitest con dependency injection usato in `src/lib/toy-ads/`, con mock espliciti delle dipendenze esterne (DB, storage) e nessun mock del database reale.

- **Unit test `getOrCreateConversation`:** idempotenza (stessa coppia toyAdId+buyerId restituisce stessa conversazione), guard buyer ≠ owner (restituisce errore se buyer è il proprietario), blocco su sessione scaduta (`getCurrentUserId` ritorna null).
- **Unit test `sendMessage`:** salvataggio corretto del messaggio, blocco se il mittente non è partecipante alla conversazione, gestione conversazione inesistente.
- **Unit test `startChatAction`:** redirect a `/dashboard/chats/[id]` su successo, gestione utente non autenticato, blocco se owner tenta di avviare chat sul proprio annuncio.
- **Test pagina chat:** verifica che un utente non-partecipante riceva 404; verifica che la card di riferimento all'annuncio sia presente nell'HTML renderizzato.
