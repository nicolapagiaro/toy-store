## User Story

**Epic:** EP-001 — Gestione Annunci Giocattoli
**Priorità:** HIGH | **Story Points:** 3

**Story**
As genitore registrato,
I want modificare dati e immagini di un mio annuncio gia pubblicato,
so that posso mantenerlo aggiornato e rilevante.

**Criteri di Accettazione**
- [ ] Posso modificare solo annunci di mia proprieta.
- [ ] Le modifiche vengono validate con le stesse regole del flusso di creazione.
- [ ] Dopo il salvataggio, la scheda annuncio mostra immediatamente i nuovi dati.

---

## Soluzione Tecnica

Per minimizzare regressioni e tempi, estendiamo il flusso gia introdotto per creazione annuncio riusando validazione e pattern server action, aggiungendo ownership check server-side e gestione foto differenziale (mantieni/rimuovi/aggiungi) prima del persist su Prisma.

- Introdurre un servizio `updateToyAdFromFormData` in `src/lib/toy-ads/service.ts` che carica l'annuncio corrente, verifica `ownerId === currentUser.id`, valida i campi con regole condivise (`validateToyAdInput`) e orchestra aggiornamento dati + immagini in modo atomico.
- Estendere `src/lib/toy-ads/validation.ts` con una variante di normalizzazione/validazione per edit che accetti foto gia presenti (`existingImagePaths`) e nuove foto (`newPhotos`), mantenendo il vincolo "almeno una foto totale" dopo le rimozioni.
- Creare server action dedicata in `src/app/dashboard/ads/[adId]/edit/actions.ts` con dipendenze Supabase Storage (upload nuove foto e delete opzionale di foto rimosse) e aggiornamento Prisma via `prisma.toyAd.update`.
- Aggiungere pagina protetta `src/app/dashboard/ads/[adId]/edit/page.tsx` che precompila i dati correnti e mostra errori field-level, bloccando accesso/modifica quando utente non proprietario (redirect o not found coerente).
- Aggiornare pagina dettaglio annuncio (`src/app/dashboard/ads/[adId]/page.tsx`, da introdurre se non ancora presente nel branch) per includere CTA "Modifica annuncio" solo per il proprietario e mostrare immediatamente i dati aggiornati dopo submit/redirect.
- Gestire concorrenza base tramite `updatedAt` check o confronto snapshot per evitare overwrite silenziosi in caso di modifiche contemporanee su stesso annuncio.

## Strategia di Test

La strategia combina unit test su validazione/orchestrazione e integration test su server action per coprire ownership, regole di business e sincronizzazione immediata del dettaglio annuncio.

- Unit: testare validazione edit in `validation.ts` (campi obbligatori, enum validi, foto totali >= 1, combinazioni mantiene/rimuove/aggiunge).
- Unit: testare servizio `updateToyAdFromFormData` con dependency injection (utente non autenticato, utente non proprietario, upload fallito, update riuscito).
- Integration: testare server action edit con DB di test/mock Prisma verificando che solo il proprietario possa aggiornare e che i dati persistiti riflettano il payload valido.
- Integration/UI: testare pagina edit precompilata e rendering errori dopo submit invalido mantenendo i valori inseriti.
- Regression: testare che il flusso di creazione annuncio (`createToyAdFromFormData`) continui a usare le stesse regole senza regressioni introdotte dal refactor condiviso.
- End-to-end smoke (manuale o automatizzato): proprietario modifica titolo/foto e al redirect il dettaglio mostra subito i nuovi dati; utente non proprietario non vede CTA edit e non puo forzare la route.
